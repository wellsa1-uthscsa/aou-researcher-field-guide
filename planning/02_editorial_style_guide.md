# The All of Us Researcher Field Guide

# Editorial Style Guide

**Version:** 1.0.0  
**Status:** Adopted  
**Effective Date:** July 17, 2026

---

# 1. Purpose

The Editorial Style Guide establishes the writing, educational, and visual conventions used throughout the All of Us Researcher Field Guide.

Its purpose is to ensure that every page, regardless of author or subject matter, provides a consistent educational experience. Readers should feel as though the entire guide was written by a single editorial team with a unified philosophy.

This document governs:

- Writing style
- Educational structure
- Technical presentation
- Terminology
- Figures
- Code examples
- Navigation
- Cross-referencing
- Review standards

---

# 2. Editorial Philosophy

The Field Guide teaches scientific reasoning rather than software operation.

Readers should leave each page understanding:

- What they are doing
- Why they are doing it
- How to implement it
- How to validate it
- When to modify the approach

Whenever possible, pages should emphasize transferable concepts over platform-specific instructions.

---

# 3. Writing Voice

The guide should read like an experienced mentor teaching a new investigator.

**The tone is:**

- Professional
- Academic
- Practical
- Encouraging
- Direct

It should never sound:

- Promotional
- Corporate
- Marketing-oriented
- Overly conversational
- Condescending

---

# 4. Writing Principles

## Explain before instructing

Always introduce the scientific purpose before describing implementation.

## Use active voice

Prefer "Query the table." over "The table is queried."

## Prefer precision

Define terminology clearly and use it consistently.

## Minimize jargon

Assume readers understand biomedical science but may be unfamiliar with cloud computing or the Researcher Workbench.

---

# 5. Reading Level

Target audience:

- Senior undergraduate
- Graduate student
- Medical student
- Clinical fellow
- Early-career investigator

---

# 6. Standard Page Structure

Every instructional page should follow this structure:

1. Title
2. Reading Time
3. Difficulty
4. Last Updated
5. Prerequisites
6. Learning Objectives
7. Why This Matters
8. Background
9. Implementation
10. Validation
11. Common Mistakes
12. Performance Tips (optional)
13. Related Guides

---

# 7. Page Length

Target reading time: **8–15 minutes**

Target length: **1,000–2,000 words**

Longer topics should be divided into multiple guides.

---

# 8. One Question Rule

Each page answers one clearly defined research question.

---

# 9. Titles

Use descriptive, searchable titles.

Examples:

- Understanding OMOP Condition Occurrence
- Building Sleep Episodes
- Constructing Polygenic Risk Scores

Avoid vague titles such as "Introduction" or "Part 1."

---

# 10. Terminology

Use official names consistently.

Examples:

- Researcher Workbench
- Dataset Builder
- BigQuery
- OMOP Common Data Model

Spell out abbreviations on first use.

---

# 11. Figures

Every major page should include at least one educational figure.

Preferred:

- Workflow diagrams
- Architecture diagrams
- Annotated screenshots
- Tables
- Concept maps

---

# 12. Code Standards

Every code example should:

- Be tested
- Be reproducible
- Include explanatory comments
- Avoid unnecessary complexity

---

# 13. Callout Standards

Only five standardized callouts should exist:

- Background
- Tip
- Validation
- Performance
- Common Mistake

---

# 14. Cross-References

Every page should include links to:

- Prerequisite guides
- Related topics
- Next recommended reading

Readers should never reach a dead end.

---

# 15. Citations

Prefer primary literature.

Use official documentation for software behavior.

---

# 16. Review Checklist

Every page should satisfy the following before publication:

- Question clearly stated
- Learning objectives present
- Scientific motivation included
- Code tested
- Validation complete
- Figures labeled
- Links functional
- Citations verified
- Formatting consistent

---

# 17. Hallmarks of a Field Guide Page

An excellent page should:

- Teach one enduring scientific idea.
- Balance scientific reasoning with implementation.
- Be immediately useful.
- Explain why the workflow works.
- Age gracefully as the platform evolves.
- Leave readers with a reusable mental model.

---

# Editorial Standard

> Could a researcher unfamiliar with this topic successfully complete the workflow, understand why each step was performed, recognize common failure modes, and adapt the methodology to their own project after reading this page?

If the answer is not unequivocally **yes**, the page is not ready for publication.
